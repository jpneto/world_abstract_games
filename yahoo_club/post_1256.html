> I think a mutator would be a game rule that makes no reference to
> equipment specific to the game. For instance, the rule specifying how
> a Knight moves on a square-grid board has no applicability to a game
> where there are no Knights or in which the board is not square-grid.
> But the "double move" mutator could apply almost anywhere.

That's an interesting view.

The idea of abstraction should be - in a yet theoretical mutator
oriented programming language for abgames - linked with the idea of
composition so that each mutator, if it had a proper type, could be
composed with other mutators to easily form games and variants. A
programmer would just need to link mutators (and eventually create new
mutators) to implement a game.

This would be a different approach that Zillions uses, focusing on the
functional aspects and recycling of simple mutators/abstract game rules.
